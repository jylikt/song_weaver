1) Почему «очень долго» (~14 минут в логе)
В логе явно:

запрошено Generating 18000 tokens (30 с × 75 fps × 8 codebooks);
при этом модель предупреждает: max length 16384, а ты просишь 18000 — то есть запрос противоречит лимиту контекста/генерации.
Такой объём autoregressive generation на GPU — это много шагов, плюс возможная деградация/странное поведение у края лимита. Отсюда и минуты ожидания.

Идея: не подбирать max_new_tokens «на глаз» по формуле — для YuE нужно повторить официальную схему из их репозитория (сколько шагов, как интерливятся кодбуки, какой лимит длины). Сейчас формула в коде, судя по логу, не соответствует тому, как YuE реально генерирует.

2) Почему фронт пишет Failed to reach worker at http://localhost:9000
Генерация на воркере заняла ~14 минут, а song-gen держит один HTTP-запрос POST /generate до конца (через httpx с REMOTE_WORKER_TIMEOUT_SEC).

Если таймаут 300 с (5 мин) — клиент оборвётся раньше, чем воркер закончит, даже если воркер ещё считает.

Идеи:

Увеличить REMOTE_WORKER_TIMEOUT_SEC в song-gen до реалистичного для длинной генерации (например 1200–3600 с) — костыль, но быстро проверить гипотезу.
Правильная архитектура: воркер сразу отвечает 202 + job_id, а клиент поллит /jobs/... — тогда нет «одного запроса на 15 минут». Это уже изменение контракта, но именно так обычно делают для тяжёлого инференса.
3) Кодек xcodec2 не загрузился
Лог:

Failed to load xcodec2 codec from 'm-a-p/xcodec2': ... is not a valid model identifier

Значит путь/идентификатор кодека неверный, репозиторий переименован/приватен/нужен токен, или нужен локальный путь после hf download.

Пока codec_loaded=False, нормального «токены → звук» не будет.

Идеи:

Найти в документации YuE точное имя кодека на HF и/или отдельную загрузку весов.
При необходимости huggingface-cli login и HF_TOKEN для приватных/гейтед репо.
Держать кодек локально и указывать путь в env.
4) Ошибка «No audio codec tokens … all below text_vocab_size»
Это ключевое: модель сгенерировала 18000 токенов, но по вашей эвристике ни один не попал в «аудио»-диапазон — всё выглядит как текстовый словарь (до 83734).

То есть текущий путь «обычный Llama + model.generate» для этого чекпоинта не совпадает с тем, как YuE ожидает формировать мультимодальные/кодековые токены (скорее всего нужен специальный шаблон промпта, спецтокены, отдельная логика шагов, как в официальном инференсе).

Идеи:

Взять официальный скрипт инференса из репозитория YuE и встроить его в воркер (или вызывать как subprocess), а не универсальный AutoModelForCausalLM.generate.
Проверить, не нужен ли trust_remote_code=True и кастомный класс модели с корректным forward/generate для аудио.
5) Что улучшить «в целом» (направления, без правок сейчас)
Проблема	Направление
Долго	Уменьшить длительность для отладки (5–10 с), поправить формулу токенов по официальному YuE
Таймаут song-gen	Увеличить timeout или асинхронный job API на воркере
Нет звука / неверные токены	Официальный пайплайн YuE + рабочий кодек
Кодек не найден	Правильный repo id / локальный путь / HF token
Краткий вывод
Сейчас узкое место не «медленный декодер WAV», а несовпадение упрощённого inference в gpu-worker с тем, как устроен YuE (кодек, формат, токены), плюс HTTP-таймаут на стороне song-gen при длинной генерации.

Имеет смысл ориентироваться на официальный quickstart из их GitHub и на корректный HF-идентификатор кодека, а не на доработку только «декодера» вокруг уже сгенерированных ID.
